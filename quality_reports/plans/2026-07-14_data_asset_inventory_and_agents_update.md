## 任务目标

依据当前工作区中的实际文件、数据结构、构建脚本、版本与血缘文档以及可追溯的既有工作记录，确认目前可用于实证分析的原始或源数据、已构建面板及其主要特征，形成一个稳定的数据资产说明入口，并在 `AGENTS.md` 中写明该入口、核心数据版本和后续操作边界。

## 范围与定义

1. “原始或源数据”包括项目内外被构建脚本直接读取、且尚未被本项目合并为主分析面板的数据文件；外部路径只记录实际可访问的文件，不复制或纳入 Git。
2. “已构建面板”包括供 Stata、R 或 Python 直接开展分析的 `.dta`、`.csv`、`.parquet` 等分析就绪文件，以及具有明确构建关系的中间面板。
3. 临时目录中的文件仅在仍然存在且能从脚本、日志或血缘文档确认用途时纳入；历史输出不得被描述为当前唯一真值。
4. 不删除、不移动、不重写任何数据、日志、历史输出或既有计划；不将数据文件加入 Git。

## 多代理分工

1. 原始数据审计：定位源文件及外部输入，记录路径、格式、大小、修改时间、变量与覆盖范围，并寻找对应构建脚本。
2. 面板数据审计：定位分析就绪面板，核验行列数、`grid_id`—`year` 键、年份、重复、缺失和核心变量可用性。
3. 文档与血缘审计：核对 `README.md`、`docs/`、`quality_reports/`、版本注册表及脚本引用，识别过时、冲突或缺少依据的描述。
4. 主流程：统一证据、处理冲突、补做统计核验、编写文档并更新 `AGENTS.md`。

## 三层核验规则

每一项写入正式说明的数据资产，至少经过以下三类检查中的两类；主分析面板和关键外部输入必须经过三类检查：

1. 文件层：路径存在、类型可读、大小与修改时间可记录，必要时计算哈希或抽样读取。
2. 结构层：使用对应格式读取器确认行列数、变量名、键唯一性、年份范围、核心变量及缺失情况。
3. 血缘层：在构建脚本、现有数据源文档、版本注册表、结果清单或运行日志中找到独立引用，并核对描述一致性。

## 交付物

1. `docs/DATA_ASSET_INVENTORY.md`：面向后续实证操作的数据资产说明文档。
2. `AGENTS.md`：新增简明的数据资产入口、当前推荐面板、版本边界和读取前核验要求。
3. 本计划文件中的最终验证记录，或另附同任务质量报告；只记录可从当前工作区复核的结论。

## 验收标准

1. 关键路径均可访问，文档中的行列数、年份范围和键检查可由脚本重新得到。
2. 原始/源数据与分析就绪面板被明确区分，外部文件、临时文件和持久文档的边界清楚。
3. 数据版本、构建关系、推荐用途和禁止混用的口径有明确说明。
4. `AGENTS.md` 不重复长篇统计，只提供稳定入口和必须遵守的操作规则。
5. 运行 `python .github/scripts/check_repository_policy.py --source index`、Markdown 链接检查、相关脚本语法检查以及 Git 状态/差异检查；若检查受既有未提交修改影响，单独说明。
